# 本地开发资源占用说明

启动本项目后若出现 **Bun**、**VmmemWSL** 进程导致内存占满、电脑卡顿甚至黑屏，可参考本说明排查与优化。

## 1. Bun 进程（约 12GB+）

**来源**：**不是本项目直接启动的**。本仓库前端使用 **npm + Vite**（见 `frontend/package.json`，无 `bun.lock`）。

- 这两个 Bun 进程通常来自 **Cursor / VSCode**：编辑器在打开含前端的仓库时，会用 Bun 跑部分 JavaScript/TypeScript 相关工具（如依赖解析、语言服务等），可能开多个进程。
- 其中一个进程占用到 **11GB+** 属于异常偏高，多为长时间运行或工具 bug 导致的内存膨胀。

**建议**：

- 开发时尽量用 **npm** 跑前端，不要在本项目里用 `bun run` / `bun install`，避免再多一层 Bun。
- 若确认是 Cursor 触发的 Bun：
  - 关闭不需要的 Cursor 窗口或工作区，只保留当前项目。
  - 用任务管理器结束占用特别大的那个 Bun 进程，必要时重启 Cursor。
- 本机内存不足时，可考虑在另一台机器或仅用 Docker 跑前端，本地只开编辑器不跑前端 dev server。

---

## 2. VmmemWSL 进程（约 3GB+）

**来源**：**Docker**。在 Windows 上执行 `docker compose up ...` 时，Docker Desktop 默认使用 **WSL 2** 作为后端，所有容器（PostgreSQL、Redis、Java、Python、前端等）都跑在 WSL 虚拟机里。**VmmemWSL** 就是这台虚拟机的内存占用。

**建议**：

1. **按需启动服务，不要一次性起满**
   - 只做前端/后端联调：只起数据库与缓存  
     `docker compose up -d postgres redis`  
     然后本地用 `mvn spring-boot:run`、`npm run dev` 等。
   - 需要完整环境时再用：  
     `docker compose up -d postgres redis backend-java backend-python frontend`
   - 开发模式（会起很多容器，占用最大）：  
     `docker compose --profile dev up ...` 尽量在内存充足时使用。

2. **限制 WSL 最大内存**（推荐）
   - 在用户目录下创建或编辑 `.wslconfig`（如 `C:\Users\你的用户名\.wslconfig`）：
   ```ini
   [wsl2]
   memory=4GB
   swap=2GB
   ```
   - 保存后，在 **PowerShell（管理员）** 执行：`wsl --shutdown`，再重新打开 WSL/Docker。
   - 这样 VmmemWSL 最多约 4GB，可避免占满物理内存。

3. **用完就停**
   - 不开发时执行：`docker compose down`，可释放 WSL 中容器占用的内存。

---

## 3. 推荐本地开发方式（减轻卡顿）

| 场景           | 建议做法 |
|----------------|----------|
| 只改前端       | `docker compose up -d postgres redis`，然后 `cd frontend && npm run dev` |
| 只改 Java      | 同上，再在 IDE 里运行 Spring Boot |
| 只改 Python AI | 同上，再本地 `cd backend-python/ai-service && python -m app.main` |
| 需要完整环境   | 内存 ≥16GB 时再用 `docker compose up -d postgres redis backend-java backend-python frontend` |
| 开发模式全栈   | 内存 ≥16GB 且需要时再用 `docker compose --profile dev up ...` |

这样可以从源头上减少 **Bun**（少依赖 Cursor 多开）和 **VmmemWSL**（少起容器、限制 WSL 内存），避免再次卡黑屏。
